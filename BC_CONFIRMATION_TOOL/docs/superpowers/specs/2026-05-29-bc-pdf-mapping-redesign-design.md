# BC 회신본 PDF → AC1~AC8 매핑 재설계

작성일: 2026-05-29
대상: `BC_CONFIRMATION_TOOL` — 은행 거래 조회서 회신본 자동 매핑

---

## 1. 배경 — 왜 다시 짜는가

회신서 PDF를 읽어 조서 칸(AC1~AC8)에 자동 입력하는 기능이 AC1(예금·유가증권)만 제대로 동작하고 AC2~AC8은 거의 비거나 부정확하다.

근본 원인은 현 `section_classifier`의 **라인 단위 키워드 추측** 방식이다.

- 표의 숫자만 있는 행(키워드 없는 행)이 직전 항목 구역에 잘못 딸려 들어간다 (state machine drift).
- "한도", "잔액" 같은 단어가 예금·대출·보증에 공통 등장 → 오분류.
- 한 행이 한 AC에만 배정 → 한 회신서가 여러 AC를 동시에 담는 실제 구조를 반영 못 함.
- 본문 파서(`parse_ac2`~`ac8`)도 `s[:40]` 통째 박기 + `_amount(s,"한도")` 앵커 방식이라 라벨 없는 표 형식에서 금액이 0으로 나온다.

## 2. 핵심 사실 (30개 회신서 전수 실측)

회신서는 라인별로 추측할 게 아니라 **번호 매긴 섹션 헤더**("1. … 다음과 같습니다 / 2. …")로 구역이 명확히 나뉜다. 헤더 문구·개수로 양식 종류를 지문처럼 식별할 수 있다.

전자 회신서 = **4개 양식 패밀리** + 우편(OCR):

| 양식 | 섹션수 | 해당 기관 |
|------|-------|----------|
| 은행형 | 8 (1·2·3·4·5·6·7·8·10) | 국민·신한·우리·KEB하나·산업·아이엠뱅크·수출입·농협 |
| 증권형 | 7 | 대신·메리츠·신한투자·KB증권·NH·삼성·미래에셋·키움·한화투자·증권금융 |
| 손보형 | 5 | KB손보·한화손보·메리츠화재·현대해상·흥국 |
| 보증보험형 | 8 | 서울보증보험 (헤더 문구 자체가 다름) |
| 우편(OCR) | 0 | 홍콩·런던 해외지점·예별·카일이삼·국민런던 6건 — 헤더 추출 실패 |

같은 "2번 섹션"이 은행형·손보형은 대출(AC2)이지만 증권형은 3번이 대출이다. **양식을 먼저 식별해야 번호→AC 매핑이 성립한다.**

## 3. 양식별 섹션 → AC 매핑표 (실측 헤더 기반)

### 은행형 (8 섹션)
| 섹션 | 헤더 요지 | → AC |
|------|----------|------|
| 1 | 금융상품(예금)의 내용 | **AC1** |
| 2 | 대출거래의 내용 | **AC2** |
| 3 | 지급보증 및 기타 약정사항 | **AC4** |
| 4 | 미결제파생상품계약 | **AC3** |
| 5 | 타 법인을 위하여 제공한 담보·연대보증 | **AC5 (회사가 제공)** + AC4 연대보증분 |
| 6 | 교부한 전자어음·어음수표 용지 | **AC6** |
| 7 | 교부한 어음·수표 중 미발행·미결제·미회수 | **AC6** |
| 8 | 담보·견질 목적 보관 어음·수표 | **AC6 (받은 어음)** |
| 10 | 당좌거래명세 | **AC6 ③ 당좌예금** |

### 증권형 (7 섹션)
| 섹션 | 헤더 요지 | → AC |
|------|----------|------|
| 1 | 유가증권 등 금융상품 | **AC1** |
| 2 | 유가증권 상세명세 | **AC1_DETAIL** (종목별) |
| 3 | 대출거래 | **AC2** |
| 4 | 지급보증 및 기타 약정 | **AC4** |
| 5 | 미결제파생상품 | **AC3** |
| 6 | 타인 위한 담보·연대보증 제공 | **AC5 + AC4** |
| 7 | 담보·견질 보관 어음·수표 | **AC6** |

### 손보형 (5 섹션)
| 섹션 | 헤더 요지 | → AC |
|------|----------|------|
| 1 | 보험거래 내용 | **AC7** |
| 2 | 대출거래 | **AC2** |
| 3 | 지급보증 및 기타 약정 | **AC4** |
| 4 | 담보·견질 보관 어음·수표 | **AC6** |
| 5 | 타인 위한 담보·연대보증 | **AC5 + AC4** |

### 보증보험형 (서울보증, 8 섹션)
| 섹션 | 헤더 요지 | → AC |
|------|----------|------|
| 1 | 제공한 보증(보험) | **AC4** |
| 2 | 신용보강 위한 신용보험 가입 | **AC7** |
| 3 | 기타 공제·보험상품 | **AC7** |
| 4 | 대출·융자·투자거래 | **AC2** |
| 5 | 출자금 등 | **AC1** (또는 AC8) |
| 6 | 한도거래 및 기타 약정 | **AC4** |
| 7 | 담보 어음·수표 | **AC6** |
| 8 | 타인 위한 담보·연대보증 | **AC5 + AC4** |

> AC4(지급보증)와 AC5(담보제공자산) 모두 ①회사가 제공받은 ②회사가 제공한 2개 sub-section을 갖는다 (V1 조서 구조). "타 법인을 위하여 제공한 담보·연대보증" 섹션은 회사가 **제공한** 쪽(②)으로 라우팅한다. 정확한 row 위치는 구현 단계에서 V1 4150 템플릿 시트를 열어 확정한다.

## 4. 설계 — 5단계 조립라인

기존 `분류 → 추출` 1패스를 `지문 → 분할 → 라우팅 → 파싱 → fallback`으로 재구성.

```
PDF
 → [1] 양식 지문 인식 (FormFingerprinter)
 → [2] 양식별 섹션↔AC 매핑표 조회 (FormProfile)
 → [3] 헤더 앵커로 구역 분할 (SectionSplitter)
 → [4] AC별 컬럼 인식 행 파서 (per-AC RowParser)
 → [5] 우편·비표준 fallback + 수동검토 플래그
 → ExtractedRecord (DB)
```

### 단위별 책임

**[1] FormFingerprinter** — `(섹션 헤더 개수, 첫 2~3개 헤더 키워드)` → 양식 패밀리 식별. 미일치 시 `"unknown"` 반환(→ fallback).
- 입력: 추출 텍스트
- 출력: `form_family: Literal["bank","securities","insurance","surety","unknown"]`

**[2] FormProfile** — 양식별 `{섹션번호: (AC, 컬럼순서, sub-section)}` 매핑표. 데이터(YAML/dict)로 분리해 코드 수정 없이 양식 추가 가능. 위 §3 표가 초기 데이터.

**[3] SectionSplitter** — 번호 헤더 정규식으로 구역 경계 탐지. 헤더~다음 헤더 사이 블록을 통째로 그 섹션에 할당. (라인별 추측 폐기 → drift 제거)
- 출력: `{section_no: block_text}`

**[4] RowParser (AC별)** — 각 블록을 AC 스키마에 맞게 파싱. AC1의 token 기반 방식(계좌/통화/날짜/금리/금액 토큰 인식)을 공통 베이스로 추출하고 AC2~8이 상속·확장.
- AC2 차입금: 한도통화·한도금액·잔액통화·대출금액·대출일·최종만기일·연이자율·최종이자지급일·상환방법·담보보증 (V1 17컬럼)
- AC4 지급보증: ①제공받은 ②제공한 2 sub
- AC5 담보: ①제공 ②받은 2 sub
- AC6 어음·수표: ①교부받은 ②교부한 ③당좌예금 잔액

**[5] Fallback** — `form_family=="unknown"`(우편 OCR 6건) 또는 헤더 0개: 기존 키워드 best-effort 파싱 유지하되 모든 레코드에 `confidence="low"` + `needs_manual_review=True`. 출력 Excel에 시각적 플래그(셀 색/주석)로 감사인 직접 확인 유도. **억지 자동화로 틀린 숫자를 넣지 않는다** (reference 회계 정확성 우선 원칙).

## 5. 데이터 흐름

```
parse_responses (UC)
  for each file:
    text = extract_or_ocr(pdf)
    family = FormFingerprinter.identify(text)
    if family == "unknown":
        recs = fallback_parse(text)           # confidence=low, manual flag
    else:
        profile = FormProfile.get(family)
        blocks = SectionSplitter.split(text, profile)
        for section_no, block in blocks:
            ac, layout = profile.route(section_no)
            recs += RowParser.for_ac(ac).parse(block, layout)
    persist(recs)
```

## 6. 에러 처리·엣지

- 헤더 일부 누락(기업은행 등 컬럼 빠짐): 양식 식별은 첫 2~3 헤더만 보므로 후반 누락에 강건. 빠진 컬럼은 해당 필드 `None`.
- 해외지점 영문 양식: 헤더 영문 → 별도 영문 프로파일 또는 `unknown` fallback.
- "해당 거래 없음" 블록: 행 0개로 정상 처리(레코드 미생성).
- 한 PDF가 여러 AC 동시: §4 구조가 기본 지원(블록별 라우팅).

## 7. 테스트 전략

- **양식 식별 테스트**: 30개 PDF 각각 기대 family와 대조 (회귀 고정).
- **섹션 분할 테스트**: 대표 4개 양식에서 섹션 개수·경계 정확성.
- **AC별 파싱 골든 테스트**: 대표 회신서 1건씩 AC2·4·5·6 손으로 검증한 기대 레코드와 대조.
- **fallback 테스트**: 우편 6건 → `needs_manual_review=True` 표시 확인.

## 8. 구현 우선순위 (합의됨)

AC2(차입금) → AC4(지급보증) → AC5(담보) → AC6(어음·수표) → 후순위(AC3 파생·AC7 보험·AC8 리스).
빈도·ROI 순. 1~3단계 공통 엔진(지문·분할·라우팅)을 먼저 세우고 AC별 RowParser를 우선순위대로 붙인다.

## 9. 범위 밖 (YAGNI)

- 후속측정·attribution·분기비교 — 절대 추가 안 함 (영구 결정).
- 화면·Excel에 "5803" 직접 노출 안 함.
- 우편 OCR 완전 자동화 — best-effort + 수동검토로 한정.
